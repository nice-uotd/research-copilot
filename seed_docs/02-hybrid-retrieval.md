# 混合检索与 RRF 融合

## 为什么要混合而不是只用向量

只用向量检索（dense retrieval）有两个常见弱点：第一，对**罕见专有名词**或**精确术语**（比如某个 SDK 名、某个具体数字）召回不稳；第二，当用户用同义词改写或跨语言提问时，向量相似度可能失效。BM25 关键词检索在这两点上恰好相反——精确术语稳但同义词失败。

混合检索（hybrid retrieval）就是同时跑向量和 BM25，再把两路结果融合，让两者互补。

## 三路实现细节

**向量检索路径**：
1. 用 `text-embedding-v3` 把用户问题编码为 1024 维向量
2. 在 Chroma 集合上做余弦相似度近邻搜索（`hnsw:space=cosine`）
3. 返回 top-K 候选

**BM25 关键词路径**：
1. 用正则 `[\w一-鿿]+` 做中英文 unicode 分词（小写化）
2. 内存 BM25 Okapi 实现，参数 k1=1.5、b=0.75
3. 平滑 IDF 计算：`log(1 + (N - df + 0.5) / (df + 0.5))`
4. 返回 top-K 候选

**RRF 融合（Reciprocal Rank Fusion）**：
- 每条候选在每路里都有一个排名，得分 = `1 / (k + rank)`，k=60 是经验常数
- 按总分（两路得分相加）降序排序
- 关键性质：**只在乎排名不在乎原始分数**，避开了 BM25（5.7 量级）和向量（0.83 量级）量纲不匹配的问题

## 为什么 RRF 而不是加权求和

加权求和需要给两路打分做归一化（min-max 或 z-score），但归一化分布会随每次查询波动。RRF 用排名作为统一坐标，鲁棒性强，且只有一个超参 k（论文经验值 60，对长尾 IR 普遍稳定）。

## 增量索引

新文档上传时，每个 chunk 分配 UUID，同时写入 Chroma 向量库和 BM25 内存索引。重启服务时，通过 SQLite 中持久化的 chunk 内容自动重建 BM25 索引（rebuild_bm25_from_db），保证零数据丢失。

## 评测维度

通过 `mode` 参数可单独启用任一路径或混合：`vector` / `keyword` / `hybrid`。配合评测脚本对同一题集横向对比 Hit@1 / Hit@5 / MRR，可以量化"为什么必须混合"。
